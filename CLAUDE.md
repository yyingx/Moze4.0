## 概述

Python 脚本集，将支付宝/微信/云闪付账单转换为 Moze 4.0 导入格式。

## 快速开始

```bash
# 依赖安装
pip install pandas numpy openpyxl
pip install paddlepaddle==2.6.2 paddleocr==2.7.3 opencv-python  # OCR 需要

# 常用命令（推荐使用 Skill）
/convert                                   # 账单转换（自动检测最新版本）
/ocr                                       # 云闪付 OCR
/reimburse                                 # 差旅报销生成
/commit                                    # 规范化 commit（分析变更）

# 或直接运行脚本
python Moze4.0_Import_v11.92.py           # 支付宝/微信账单转换
python "Moze4.0_云闪付PaddleOCR 稳定版_.py"  # 云闪付 OCR
python Moze4.0报销_Enhanced.py             # 差旅报销生成
```

## 核心数据结构

**⚠️ 修改前必读：**

- **`CONFIG`** - 内部转账对象和默认账户配置；v11.92 起用 `WECHAT_TARGET_*`、`WECHAT_BOUND_CARD_*`、`WUHANTONG_RECHARGE_MERCHANT`、`MOZE_ACCOUNT_*` 区分原始交易对方和 Moze 账户
- **`DATA_SOURCE`** - 关键词字典（不要随便改）
- **`INGREDIENT_PRIORITY`** - 匹配优先级顺序（顺序决定匹配结果）
- **`RAW_MAPPING_CONFIG`** - 子类别 → (记录类型, 主类别, 项目) 映射
- **`SUBCAT_KEYWORDS`** - 备注中的子类别关键词，当前 59 个输入关键词
- **`NAME_KEYWORDS`** - 备注中的名称关键词，当前 15 个输入关键词

**餐饮规则边界：**
- `DATA_SOURCE['MEAL']` 是餐别触发器，可包含固定餐饮地点、品牌、食堂和具体餐品词。
- `Moze Dict.xlsx` 负责商家清洗、商家+商品复合规则、描述和标签。
- 复合规则示例：`商家(old)=武汉工程大学` + `商品=武汉工程大学`，或 `商家(old)=湖北工业大学` + `商品=东苑`。
- 需要按时间推导餐别的字典规则，`子类别` 应留空，`项目` 填 `食`；同时确保对应地点/品牌/餐品能命中 `MEAL`。
- `商品` 列支持正则，但必须设置 `is_regex=1`；如需同时匹配东苑/西苑可写 `(?:东|西)苑`。

**修改禁忌：**
- 不要打乱 `INGREDIENT_PRIORITY` 顺序
- 不要删除 `RAW_MAPPING_CONFIG` 中已有的子类别
- 不要在 `exclude_if_match` 中引用不存在的 key
- 不要直接改旧版本脚本；需要行为变更时新建 `Moze4.0_Import_v11.XX.py`
- 新建版本脚本时必须同步更新 `__version__` 和 `__updated__`
- 不要提交支付宝/微信源账单、Moze 导出 CSV、截图账单等私人数据

**字段语义：**
- `名称` = 子类别下的结构化明细；普通支出/收入和报销记录共用 Moze 的 `名称` 字段。
- `对象` = 资金往来的对象，主要用于应收应付、报销、借贷等记录。
- `描述` = 自由补充说明。
- 字段层面共用 `名称`，但规则层面不要合并：普通消费由 `NAME_KEYWORDS`/`DATA_SOURCE` 推导，报销费用由 `REIM_TRAVEL_KEYS`/`REIM_EXPENSE_KEYS` 单独写入 `名称`。

## 当前导入规则要点

- 支出默认导入；收入只导入明确可分类记录，避免把退款、内部流转误作收入。
- `已全额退款` 和 `交易关闭` 不导入。
- 退款行只提示不导入：
  - 支付宝按订单号前 8 位推断原交易日期，`0-1` 天视为快速退款，`>=2` 天视为退货/部分退款。
  - 微信按状态识别：`已退款¥x` 作为需要手动处理的退款收入；`已退款(¥x)` 的原支出行不单独提示。
  - 退款提示使用 CSV 格式输出，并带 `来源` 列。
- 无备注微信转账收入只提示不导入：
  - 条件：微信、收入、交易类型为转账、备注为空、商品文本包含“微信转账”。
  - 排除已定义的内部转账对象，如肖恩、YingCH、工商银行(9579)、上海雪球数智科技有限公司。
  - 有业务备注时备注规则优先，不进入该提醒。
- 内部转账规则：
  - 肖恩：收入按 `零钱2 -> 零钱3`，支出按 `零钱3 -> 零钱2`。
  - YingCH：收入按 `零钱1 -> 零钱3`，支出按 `零钱3 -> 零钱1`。
  - 工商银行(9579)：零钱提现按 `零钱3 -> 工商银行`，零钱充值按 `工商银行 -> 零钱3`。
  - 上海雪球数智科技有限公司是武汉通充值商户，按 `平安银行4946 -> 武汉通` 处理。
- `停车费` 与 `叉车费` 同属公司费用报销；备注写 `停车费...` 时按 `报账/#费用报销/对象=天之逸` 处理。
- 拼多多等返利广告描述会清空，避免广告文案进入 Moze 描述。
- 手动备注和字典明确写入的子类别优先；食材/水果等启发式关键词只补空子类别，不覆盖已有分类。
- `PAUSE_ON_EXIT = False`，脚本运行结束默认不等待 Enter。
