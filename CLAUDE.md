# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## 概述

本仓库包含 Python 脚本，用于将支付宝、微信、云闪付账单转换为 Moze 4.0 导入格式，实现自动记账。

- **支付宝/微信**: 解析 CSV/XLSX 文件，基于规则字典和关键词智能分类
- **云闪付**: 使用 PaddleOCR 识别账单截图，智能提取交易信息

## 运行脚本

## 架构

### 主脚本处理流程

所有版本（v11.67+）遵循统一的处理管道：

1. **数据加载**
   - `load_settings()` / `load_rules()` - 从 `Moze Dict.xlsx` 加载配置和规则
   - `sniff_and_load_data()` - 动态检测 CSV header 位置，自动识别支付宝/微信格式

2. **预处理**
   - `infer_income_expense()` - 根据备注关键词自动推断收/支类型
   - `standardize_accounts()` - 标准化账户名称（正则匹配）

3. **分类处理**（优先级从高到低）
   - `process_transfers()` - 处理转账交易（包括闭环验证）
   - `process_main()` - 字典规则匹配（精确匹配 → 正则匹配）
   - `process_heuristics()` - 关键词启发式分类（基于 `INGREDIENT_CONFIG`）
   - `apply_meal_time_logic()` - 正餐时间推导（早/午/晚/夜宵）

4. **后处理**
   - 点号分隔解析（`对象.描述` 格式）
   - 应收应付款项处理
   - 淘宝订单识别（T200P 前缀）

5. **输出**
   - `save_result()` - 输出带时间戳的 CSV 到 `Moze4.0_Import/`

### 核心数据结构

**⚠️ 关键约束 - 修改前必读：**

#### v11.70+ (JSON 配置版)

配置存储在 `ingredient_config.json`：

- **`INGREDIENT_CONFIG`** - 字典结构，包含每个分类的：
  - `keywords`: 关键词列表
  - `name`: 显示名称
  - `subcategory`: 子类别
  - `priority`: 匹配优先级（数字越小越先匹配）
  - `exclude_if_match`: 互斥规则（如 PORK 排除 COOKED）

- **`RAW_MAPPING_CONFIG`** - 子类别到 `(记录类型, 主类别, 项目)` 的映射列表

**修改 JSON 后无需重构脚本，直接运行即可。**

#### v11.67-v11.71 (单文件版)

数据结构硬编码在脚本中：

- **`DATA_SOURCE`** (旧) / **`INGREDIENT_CONFIG`** (新) - 关键词字典
- **`INGREDIENT_PRIORITY`** - 处理顺序列表 `[(key, name, subcategory), ...]`
- **`RAW_MAPPING_CONFIG`** - 子类别映射配置
- **`AUTO_MAP_DICT`** - 运行时从 `RAW_MAPPING_CONFIG` 自动生成

**修改规则：**
1. `INGREDIENT_CONFIG` 的 key 必须与 `INGREDIENT_PRIORITY` 的 key 对应
2. `INGREDIENT_PRIORITY` 的顺序决定匹配优先级（先出现 = 优先匹配）
3. `exclude_if_match` 机制允许跨类别互斥（如"猪肉"不能同时是"熟食"）

## 分类优先级

处理顺序（从高到低）：

1. **备注驱动** - 用户在备注中标注，如 `探索claude` → 子类=探索，描述=claude
2. **字典规则** (`Moze Dict.xlsx`) - 先精确匹配商家名，再正则匹配
3. **关键词检测** (`INGREDIENT_CONFIG`) - 按 `priority` 顺序匹配关键词
4. **时间推导餐饮** - 检测为正餐但无子类别时，根据交易时间判断早餐/午餐/晚餐/夜宵

**关键词互斥逻辑：**
- 已匹配高优先级类别 → 跳过低优先级
- `exclude_if_match` 生效 → 跳过互斥类别（如猪肉不匹配熟食）

### 淘宝订单识别

通过商户单号前缀 `T200P` 识别淘宝订单（兼容 T200P4、T200P5 等）。

### 转账处理

通过 `CONFIG['TRANSFER_TARGET_x']` 配置转账对方，自动生成转入/转出配对记录。

### 手动增加行（v11.67 新增）

在支付宝/微信账单 Excel 中手动添加交易记录时，**可以不填"收/支"列**，系统会根据备注关键词自动推断：

| 备注格式 | 自动推断为 | 说明 |
|---------|----------|------|
| `借入xxx` | 收入 | 应付款项（收钱） |
| `借出xxx` | 支出 | 应收款项（借出钱） |
| `代付xxx` | 支出 | 应收款项（代付款） |
| `报账xxx` | 支出 | 应收款项（公司报销） |
| `押金xxx` | 支出 | 应收款项（押金） |
| `材料费xxx` | 支出 | 应收款项（费用报销） |
| `物流运输xxx` | 支出 | 应收款项（费用报销） |

**使用场景：** 现金交易、未在支付宝/微信记录的交易等。

### 应收应付款项逻辑

**应收款项**（记录类型 = `应收款项`）
- **子类别**: 报账、借出、代付、押金
- **金额**: 负数（支出）
- **对象**: 必填（借给谁、代付给谁）
- **商家/项目**: 自动清空
- **触发**: 备注以关键词开头 + 收/支=支出

**应付款项**（记录类型 = `应付款项`）
- **子类别**: 借入
- **金额**: 正数（收入）
- **对象**: 必填（从谁借入）
- **商家/项目**: 自动清空
- **触发**: 备注以"借入"开头 + 收/支=收入

**点号分隔格式**（v11.67 新增）：
- 在应收应付类别中，支持 `对象.描述` 格式
- 示例：`借入谢辉.钉子` → 对象=谢辉，描述=钉子

## 报销生成器工作原理

### 双向分录机制

每笔补贴自动生成两条记录：

1. **收入记录** - 收到公司补贴
   - 记录类型：收入
   - 主类别：收入
   - 子类别：福利补贴
   - 金额：正数（如 +40 元）
   - 项目：工作
   - 商家：天之逸

2. **应收款项** - 冲抵待报账金额
   - 记录类型：应收款项
   - 主类别：应收款项
   - 子类别：报账
   - 金额：负数（如 -40 元）
   - 对象：天之逸
   - 标签：#差旅费报销

**财务意义：**
- 第一笔记录你的实际收入（钱包增加）
- 第二笔记录公司欠你的报销款减少（应收减少）
- 两笔配对后净资产不变，但准确反映收支流水

### 补贴类型

| 类型 | 名称 | 默认金额 | 计算方式 |
|------|------|----------|----------|
| 餐补 | 餐费补贴 | 40元/天 | 固定金额 × 天数 |
| 交补 | 交通补贴 | 20元/天 | 固定金额 × 天数 |
| 住宿 | 住宿补贴 | 140元/天 | 标准 - 实际花费（仅差额）|

**时间戳规则：**
- 所有记录使用固定时间 17:30:00（避免与真实交易冲突）
- 同一天的餐补和交补间隔10分钟
- 收入与应收款项间隔1秒（保持配对关系）

## 云闪付 OCR 特性

### 智能识别
- **日期推导**: 自动识别月-日格式，跨年时自动调整年份
- **金额提取**: 处理 OCR 误识别（l/I/| → 1），合并多段数字
- **账户映射**: 支持尾号匹配 + 特殊账户名（活期+、零钱等）
- **时间修正**: 自动修正常见 OCR 错误（如 71 → 11）

### 自动分类逻辑
- **转账识别**: "转给" + 本人姓名 → 转出/转入
- **借贷识别**: "转给/来自" + 他人姓名 → 借出/借入（应收/应付款项）
- **还款识别**: "还款" 关键词 → 信用卡还款

### 验证机制
1. **闭环验证**: 检查转账配对（金额相反、日期相同），自动同步时间
2. **时间检查**: 提示缺失时间的记录，建议补充
3. **对象清理**: 自动去除"转账"、"付款"等后缀，提取纯净人名

## 常见问题

### 支付宝/微信脚本

**Q: 提示"找不到 header 行"**
- **原因**: CSV/XLSX 格式变化，header 不在默认范围
- **解决**:
  - v11.71 会自动扩展搜索范围
  - 手动调整 `ALIPAY_HEADER_RANGE` / `WECHAT_HEADER_RANGE`

**Q: 日期/金额转换失败**
- **症状**: 警告日志 "日期转换失败" / "金额转换失败"
- **解决**:
  - 检查原始文件中对应行的数据格式
  - 可能是手动添加行时未按正确格式填写

**Q: 分类不准确**
- **诊断思路**:
  1. 检查 `Moze Dict.xlsx` 是否有匹配规则
  2. 检查关键词是否在 `INGREDIENT_CONFIG` 中
  3. 检查是否被 `exclude_if_match` 排除
  4. 查看日志确认匹配路径

**Q: 转账记录未配对**
- **原因**: 转账对方名称不匹配 `CONFIG['TRANSFER_TARGET_x']`
- **解决**: 在 CONFIG 中添加新的转账目标

### 云闪付 OCR

**Q: 账户名映射错误**
- **解决**: 更新脚本中的 `ACCOUNT_MAP` 字典，添加新的尾号映射

### 报销生成器

**Q: 日期格式错误**
- **症状**: 提示 "格式应为 yymmdd"
- **解决**:
  - 单日: `250901` (2025年9月1日)
  - 日期范围: `250901-250905` (2025年9月1日至5日)
  - 注意使用两位年份

**Q: 如何修改补贴标准？**
- **解决**: 编辑脚本中的 `CFG` 字典
  ```python
  CFG = {
      'MEAL': 50,    # 修改餐补为50元/天
      'TRANS': 30,   # 修改交补为30元/天
      'HOTEL': 200,  # 修改住宿标准为200元/天
  }
  ```

**Q: 住宿补贴如何计算？**
- **原理**: 输入实际花费，自动计算差额
- **示例**: 标准140元，实际花费120元 → 生成20元补贴
- **注意**: 若实际花费超标准，无补贴记录

**Q: 生成的记录在哪里？**
- **位置**: `Moze4.0_Import/报销_YYYYMMDD_HHMMSS.csv`
- **导入**: 使用 Moze 4.0 App 导入该 CSV 文件

### Git 同步工具

**Q: 如何快速提交代码？**
- **工具**: 使用 `Auto.py` 脚本
- **运行**: `python Auto.py`
- **技巧**:
  - 输入符合规范的消息（如 `新增 v11.72: 添加功能`）
  - 直接回车快速同步（使用时间戳）
  - 会自动推送到 GitHub 和 Gitee 双仓库

**Q: Auto.py 卡住不动？**
- **原因**: 可能存在合并冲突或网络问题
- **解决**:
  - Ctrl+C 退出脚本
  - 手动执行 `git status` 查看状态
  - 解决冲突后重新运行 `Auto.py`

**Q: 想撤销上次 commit？**
- **解决**:
  ```bash
  git reset --soft HEAD~1  # 撤销 commit，保留更改
  git reset --hard HEAD~1  # 撤销 commit，丢弃更改（谨慎！）
  ```
